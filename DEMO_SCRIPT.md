# LightGuard Demo Script
## Tadhamon Smart City IDS — Attack → Detect → Alert → Log

> **Note for evaluators:** All attack traffic in this demo is generated by the built-in
> Scenario Engine (clearly labelled `is_simulation=True` in the database and logs).
> This is standard practice in IDS demonstrations and red team training exercises.
> Setting `MOCK_MODE=false` in `config/lightguard.env` enables live Scapy packet
> capture on the real network interface.

## MFA Setup Demo (Optional)
**Demonstrates:** TOTP-based MFA available on the system.

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | While logged in as admin, call: `POST /auth/mfa/setup` | Returns QR code PNG (base64) + provisioning URI |
| 2 | Scan QR in Google Authenticator or Authy | 6-digit code appears |
| 3 | Call `POST /auth/mfa/verify` with the code | `{"status": "verified"}` |
| 4 | Log out → Log in again | Step 1: password → returns `mfa_required: true` + temp_token |
| 5 | Call `POST /auth/login/verify` with temp_token + TOTP | Full access_token issued |

> **Note:** MFA setup is optional per user. Accounts without a configured TOTP secret
> proceed directly to full token issuance on password verification (backward-compatible).
> MFA enforcement (requiring all ADMIN accounts to configure TOTP) is the immediate
> next development step as documented in Chapter VI Section 6.4.

## Key Points to Highlight

1. **Real detection engine** — Scapy analyzes actual packets (port, flags, rate thresholds)
2. **RandomForest ML** — trained on NSL-KDD (99.2% offline accuracy); live feature extraction is the next development milestone
3. **Fog Computing** — 3 distributed zones with real CPU/RAM metrics; physical Raspberry Pi deployment is the next hardware milestone
4. **Adaptive Optimizer** — self-tunes anomaly threshold every 30 minutes based on analyst false-positive feedback
5. **CVE intelligence** — real CVE identifiers (CVE-2021-36260 CVSS 9.8) for actionable device-specific security guidance
6. **MFA** — TOTP endpoints implemented (`/auth/mfa/setup`, `/auth/mfa/verify`, `/auth/login/verify`); mandatory enforcement in next phase
7. **TLS/HTTPS** — run `./scripts/start_https.sh` for SSL mode in production

## Answering "Is this real or simulated?"

> "The detection engine is real — it uses Scapy to analyze actual network packets using
> six signature rules with exact thresholds (for example, ≥15 TCP SYN packets in 5 seconds
> triggers a Port Scan alert) and a trained RandomForest model for anomaly scoring.
> The attack traffic in this demo is generated by the Scenario Engine, which is clearly
> labelled `is_simulation=True` in all database records and logs — this is standard
> practice in IDS demonstrations and red team training exercises. In a production
> deployment, `MOCK_MODE=false` in `config/lightguard.env` enables live interface
> capture via Scapy."

---

## Answering "What are the limitations?"

> "LightGuard is a prototype. Six specific limitations are documented in Chapter VI:
> (1) ML live feature extraction not yet connected to Scapy capture stream;
> (2) MFA enforcement not yet mandatory (endpoints implemented, enforcement next phase);
> (3) Snort/Suricata log parsing is simplified;
> (4) Fog nodes are software-simulated — physical Raspberry Pi deployment is planned;
> (5) VLAN segmentation is logically modelled — physical 802.1Q switch not tested;
> (6) No automated threat response — iptables blocking designed but not yet connected.
> Each limitation has a specific implementation path documented in the report."
